# vera_rag_service

RAG Service — сервис семантического поиска по базе знаний для AI-ассистента «Вера» (консультант по правам людей с инвалидностью в сфере трудоустройства, проект «Работа для всех»).

## Роль в системе

Один из трёх сервисов архитектуры ассистента: **Agent Service** (оркестратор диалога) → **MCP Tools Server** (прослойка инструментов) → **RAG Service** (этот репозиторий). RAG Service принимает HTTP-запросы от MCP Tools Server и не взаимодействует с пользователем напрямую — он не знает ни про диалог, ни про Agent Service.

База знаний: нормативные документы (ТК РФ, ФЗ-181 «О социальной защите инвалидов», подзаконные акты, судебная практика) и авторские статьи-систематизации. Контент готовит и загружает Expert через `/admin` (см. `ADMIN_GUIDE.md`).

## Как это работает

### 1. Ingestion — загрузка документа в базу знаний

Запускается через `POST /api/v1/ingest` или форму `/admin/document-upload`. Один вызов = один документ целиком, пайплайн полностью синхронный (ответ возвращается только после завершения всех шагов):

1. **Извлечение текста** (`app/ingestion/extract.py`) — если документ пришёл как файл (через админку), сначала декодируется в обычный текст: `.txt`/`.md` — прямое декодирование UTF-8, `.pdf` — через `pdfplumber`, `.docx` — через `python-docx` (параграфы, гиперссылки, таблицы, колонтитулы, в исходном порядке документа; текст с картинок внутри `.docx` не извлекается). `POST /api/v1/ingest` принимает уже готовый текст (`raw_text`), минуя этот шаг.
2. **Препроцессинг** (`app/ingestion/preprocess.py`) — чистка артефактов форматирования (переносы по дефису, колонтитулы-номера страниц) и разбор на секции: для нормативных актов — по найденным `Статья N`, для авторских статей — по markdown-заголовкам.
3. **Чанкинг** (`app/ingestion/chunking.py`) — каждая секция режется на чанки по 300–500 токенов с overlap, чтобы не терять контекст на границах. `chunk_id` детерминированный (вычисляется из `document_id`+`version`+номера чанка) — повторная загрузка той же версии документа перезаписывает те же чанки, не создаёт дублей.
4. **Обогащение LLM** (`app/ingestion/enrichment.py`, Yandex Cloud) — для каждого чанка генерируется синтетический заголовок и 3–5 гипотетических вопросов, на которые этот чанк отвечает. Это компенсирует слабость векторного поиска на «сухом» юридическом тексте — реальные вопросы пользователей формулируются не так, как звучит норма закона.
5. **Эмбеддинг** (`app/embeddings/embedder.py`, Yandex Text Embeddings v2) — векторизуется текст «заголовок + чанк» и каждый гипотетический вопрос отдельно.
6. **Запись в Qdrant** — один чанк = одна точка с несколькими именованными векторами (`chunk` + до 5 `question_N`) и общим payload (текст, метаданные, заголовок, вопросы). Параллельно пишется строка в реестр `documents` (Postgres) — для истории версий и отображения в админке.
7. **Замена старой версии** — если документ с этим `document_id` уже был проиндексирован под другой `version`, старая версия удаляется из Qdrant **только после** успешной загрузки новой — без окна, когда документ временно недоступен поиску.

### 2. Search — поиск по базе знаний

Запускается через `POST /api/v1/search` или форму `/admin/search-test`:

1. **Расширение запроса** (`app/search/query_expansion.py`, LLM через Polza AI/Gemini) — перед поиском составной вопрос декомпозируется на независимые подвопросы (не более трёх), каждый переформулируется ближе к терминологии трудового права. Цель: повысить recall — пользователь формулирует «у меня инвалидность, дадут ли отпуск», нужная норма говорит про «дополнительный отпуск работникам с инвалидностью». При недоступности LLM — деградация до одного исходного запроса.
2. **Эмбеддинг запросов** — каждый вариант запроса векторизуется (Yandex, query-модель).
3. **Гибридный поиск** (`app/search/hybrid.py`) — для каждого варианта параллельно выполняется dense-поиск (косинусное сходство) и sparse-поиск (нативные sparse-векторы Qdrant с IDF, BM25-подобное ранжирование — для точных терминов вроде «статья 21», которые векторный поиск может смазать). Поиск ведётся только по актуальным редакциям (`is_actual=True`).
   - Если в запросе **не** указана `category` — поиск идёт по каждой из 5 категорий источников отдельно (категорийно-сбалансированный retrieval), чтобы крупный корпус ТК РФ не вымывал из выдачи малочисленные, но юридически значимые источники (судебную практику, авторские комментарии).
   - Если `category` указана явно — обычный плоский поиск внутри неё.
4. **RRF fusion** (`app/search/fusion.py`) — все dense- и sparse-ранжирования по всем вариантам запроса объединяются в единый список кандидатов (Reciprocal Rank Fusion).
5. **Reranking** (`app/search/reranker.py`, LLM через Polza AI/Gemini) — кандидаты переранжируются LLM по реальной смысловой релевантности запросу. Если LLM считает, что ни один кандидат не отвечает на вопрос — возвращает пустой список (сигнал «нет ответа»), не выбирает «наименее нерелевантного». При недоступности LLM — деградация до исходного порядка RRF.
6. **Лог** — каждый запрос (варианты запроса, промежуточные кандидаты, latency по шагам) пишется в `search_logs` (Postgres) — для анализа качества поиска через `/admin/dashboard`.

### 3. Админка (`/admin`, sqladmin)

Полноценная консоль управления сервисом — загрузка/просмотр/удаление документов, интерактивное тестирование поиска, дашборд состояния. Подробная инструкция — [`ADMIN_GUIDE.md`](ADMIN_GUIDE.md).

### 4. Наблюдаемость

- **Логи** — со сквозным `request_id` (один на HTTP-запрос, виден и в логах, и в `search_logs`, и в заголовке ответа `X-Request-ID`).
- **`GET /metrics`** (Prometheus) — latency/error rate по эндпоинтам.
- **`/admin/dashboard`** — состояние Postgres/Qdrant, число документов/чанков, средняя latency поиска за 24 часа, сверка реестра документов с фактическим содержимым Qdrant.

## Стек

FastAPI · Qdrant (self-hosted, векторное хранилище, named + sparse векторы) · Postgres (реестр документов + журнал поисковых запросов) · Yandex Cloud (Text Embeddings v2 + LLM-обогащение чанков) · Polza AI/Gemini (LLM-reranker) · sqladmin (админка) · slowapi (rate limiting) · Prometheus (метрики) · Alembic (миграции) · hypercorn (ASGI-сервер)

## API

Базовый префикс — `/api/v1`. Все эндпоинты, кроме `/health` и `/metrics`, требуют заголовок **`X-API-Key`** (значение — из `Settings.app.api_key`/переменной окружения `API_KEY`). Неверный или отсутствующий ключ — `401`/`422`. Каждый публичный эндпоинт ограничен по частоте запросов (`slowapi`, по IP) — при превышении лимита ответ `429`.

### `GET /api/v1/health`

Без авторизации — для health-check оркестратора. Без лимита частоты.

**Ответ `200`:**
```json
{ "status": "ok", "database": "ok" }
```
**Ответ `503`** — если Postgres недоступен: `{ "detail": "База данных недоступна." }`.

### `POST /api/v1/search`

Семантический поиск по базе знаний. Лимит — **60 запросов/мин** с одного IP.

**Тело запроса:**

| Поле | Тип | Обязательное | Описание |
|---|---|---|---|
| `query` | `string` | да | Текст поискового запроса (минимум 1 символ). |
| `audience` | `"seeker" \| "employer" \| "both"` | нет | Фильтр по целевой аудитории. `both` подходит под фильтр любой конкретной аудитории. |
| `topic` | `string` | нет | Фильтр по теме — чанк проходит, если эта тема есть в его `topics` (см. `POST /ingest`). Осмысленно только для чанков категорий `other_npa`/`case_law`/`authorial` — у `labor_code`/`federal_law` `topics` всегда пуст. |
| `category` | `"labor_code" \| "case_law" \| "federal_law" \| "other_npa" \| "authorial"` | нет | Фильтр по категории источника. Если не указан — категорийно-сбалансированный поиск по всем пяти. |
| `top_k` | `int` | нет (по умолчанию `5`) | Сколько чанков вернуть после переранжирования, 1–20. |

```json
{
  "query": "Какая квота на трудоустройство инвалидов?",
  "audience": "employer",
  "top_k": 5
}
```

**Ответ `200`:**
```json
{
  "chunks": [
    {
      "chunk_id": "uuid",
      "text": "Текст чанка.",
      "synthetic_title": "Заголовок, сгенерированный LLM",
      "source_title": "ФЗ-181, Статья 21",
      "audience": "both",
      "topics": ["квотирование"],
      "category": "federal_law",
      "section_number": "21",
      "section_title": "Статья 21. Права инвалидов в области занятости",
      "score": 0.0123
    }
  ]
}
```
`chunks` может быть пустым списком, если reranker посчитал всех кандидатов нерелевантными — это не ошибка, сигнал «нет ответа на этот вопрос». `section_number`/`section_title` присутствуют для нормативных актов с явной структурой статей, `null` для остальных категорий.

**Ошибки:** `422` — невалидное тело запроса; `500` — недоступен Embedding API или LLM-reranker после исчерпания retry (`detail` содержит подробности); `429` — превышен лимит частоты запросов.

### `POST /api/v1/ingest`

Индексация документа в базе знаний (полный пайплайн — см. раздел «Ingestion» выше). Лимит — **10 запросов/мин** с одного IP (платная и редкая операция, строже, чем `/search`).

**Тело запроса:**

| Поле | Тип | Обязательное | Описание |
|---|---|---|---|
| `document_id` | `string` | да | Идентификатор документа. Тот же `document_id` с новой `version` — обновление документа, не дубликат. |
| `category` | `Category` (см. выше) | да | Категория источника. |
| `raw_text` | `string` | да | Текст документа (уже декодированный — PDF/DOCX/MD/TXT декодирует только админка). До 1 000 000 символов. |
| `source_title` | `string` | да | Человекочитаемое название источника. |
| `audience` | `"seeker" \| "employer" \| "both"` | да | Целевая аудитория. |
| `topics` | `string[]` | нет | Темы документа. Допустимы только для `other_npa`/`case_law`/`authorial` (справочник — `/admin/topic`) — для `labor_code`/`federal_law` список должен быть пустым, иначе `422`. |
| `version` | `string` | да | Дата редакции документа, например `2026-01-01`. |
| `effective_date` | `date` | да | Дата вступления редакции в силу. |

**Ответ `200`:**
```json
{
  "document_id": "fz-181-art21",
  "version": "2026-01-01",
  "chunks_count": 3,
  "replaced_versions": [],
  "not_removed_versions": []
}
```
`replaced_versions` — версии, успешно замещённые этим вызовом. `not_removed_versions` — версии, которые должны были быть удалены, но не получилось (нужна ручная чистка через админку) — само по себе не ошибка вызова, новая версия уже проиндексирована успешно.

**Ошибки:** `422` — невалидная `category`/пустой текст/`raw_text` превышает лимит длины/документ дал больше 2000 чанков; `500` — недоступен LLM-обогащение или Embedding API после исчерпания retry; `429` — превышен лимит частоты запросов.

### `DELETE /api/v1/document/{document_id}`

Удаляет документ — **все его версии** — и из Qdrant, и из реестра `documents`. Лимит — **20 запросов/мин** с одного IP.

**Ответ `200`:**
```json
{ "document_id": "fz-181-art21" }
```
Идемпотентно — повторный вызов на уже удалённый документ тоже вернёт `200`.

### `PUT /api/v1/document/{document_id}/sections/{section_number}`

Гранулярное обновление одной статьи/пункта нормативного акта без переиндексации всего документа. Поддерживаемые категории: `labor_code`, `federal_law`, `other_npa`. Лимит — **10 запросов/мин** с одного IP.

Старые чанки секции **не удаляются** — помечаются `is_actual=False` с датой `effective_until`. Поиск их больше не видит, но данные хранятся для будущих запросов вида «какая норма действовала на дату X».

**Тело запроса:**

| Поле | Тип | Описание |
|---|---|---|
| `category` | `Category` | Категория документа. |
| `raw_text` | `string` | Текст новой редакции статьи/пункта. |
| `section_title` | `string` | Заголовок статьи/пункта (например, `"Статья 128. Отпуск без сохранения заработной платы"`). |
| `version` | `string` | Дата новой редакции, например `"2026-01-01"`. |
| `effective_date` | `date` | Дата вступления новой редакции в силу. |
| `source_title` | `string` | Название документа. |
| `audience` | `"seeker" \| "employer" \| "both"` | Целевая аудитория. |
| `topics` | `string[]` | Темы — допустимы только для `other_npa` (единственная из `SECTION_UPDATE_ALLOWED_CATEGORIES`, где темы осмысленны). |

**Ответ `200`:**
```json
{
  "document_id": "tk-rf",
  "section_number": "128",
  "parent_id": "tk-rf:128",
  "version": "2026-01-01",
  "chunks_count": 2,
  "superseded_chunks": 3
}
```
`chunks_count` — число новых чанков. `superseded_chunks` — сколько чанков старой редакции помечено неактуальными.

**Ошибки:** `422` — если `category` не поддерживает гранулярное обновление (`case_law`/`authorial`).

### `GET /metrics`

Метрики Prometheus, без авторизации (скрейперы Prometheus обычно не передают кастомные заголовки) — в production ограничивать доступ на уровне сети/firewall, не полагаться на отсутствие `X-API-Key` как на защиту.

### `/admin/*`

Веб-интерфейс управления (логин/пароль, не `X-API-Key`) — см. [`ADMIN_GUIDE.md`](ADMIN_GUIDE.md).

## Запуск локально

Краткая версия — подробности и пример заполнения `.env` см. в [`ADMIN_GUIDE.md`](ADMIN_GUIDE.md#как-локально-запустить-проект).

```bash
cp .env.example .env
# заполнить .env — минимум POSTGRES_PASSWORD, SECRET_KEY, ADMIN_LOGIN/ADMIN_PASSWORD, API_KEY

docker compose up -d db qdrant      # Postgres + Qdrant

python -m venv venv
venv\Scripts\activate                # Windows; source venv/bin/activate — Linux/macOS
pip install -r requirements-dev.txt
alembic upgrade head
hypercorn app.main:app --bind 0.0.0.0:8000 --reload
```

Проверка: `curl http://localhost:8000/api/v1/health` → `{"status":"ok","database":"ok"}`.

Альтернатива — весь сервис тоже в Docker: `docker compose up -d --build`.

## Тестирование

```bash
pytest tests/                        # обычный набор (юнит + API + интеграционные на реальных Qdrant/Postgres)
pytest -m slow tests/performance     # нагрузочный тест на синтетическом корпусе 5000 чанков, не входит в обычный прогон
ruff check .                         # линтер
```

Интеграционные тесты используют реальный локальный Qdrant (`docker-compose up -d qdrant`) и Postgres через `testcontainers` (поднимается автоматически, нужен запущенный Docker).

## Чеклист перед production-развёртыванием

Операционные пункты, не закрываемые правкой кода этого репозитория (подробности — `RAG_SERVICE_PLAN.md`, раздел 7):

- **`YANDEX_LLM_MODEL`** — указать конкретную версионированную модель, не `/rc`-канал (release candidate может обновляться без предупреждения и менять поведение, под которое подобраны обходные пути в `LlmClient`).
- **Снапшоты Qdrant** — настроить периодический вызов `POST /collections/{name}/snapshots` с выгрузкой в отдельное хранилище вне `qdrant_data`-volume — без этого инцидент с диском/контейнером означает полную потерю проиндексированного корпуса и повторную оплату всех LLM/embedding вызовов.
- **`ADMIN_SESSION_HTTPS_ONLY=true`** — обязательно за HTTPS-терминирующим reverse-proxy (по умолчанию `false` для локальной разработки).
- **`/metrics`** — ограничить доступ на уровне сети/firewall, эндпоинт не защищён `X-API-Key`.
- **Лимит памяти Qdrant** — в `docker-compose.yml` уже задан (`4GB`) — пересмотреть при росте корпуса на порядок.
