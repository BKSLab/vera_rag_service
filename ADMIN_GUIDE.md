# Инструкция: управление базой знаний через /admin

Короткая практическая инструкция для Expert/контент-менеджера — без необходимости знать код. Технические детали и архитектура — в `RAG_SERVICE_PLAN.md`, описание API — в `README.md`.

## Как локально запустить проект

Нужно для тестирования через админку на своей машине (не в общем тестовом окружении). Понадобится Docker и Python 3.12.

1. **Склонировать репозиторий и подготовить `.env`:**
   ```bash
   cp .env.example .env
   ```
   Откройте `.env` и заполните как минимум: `POSTGRES_PASSWORD`, `SECRET_KEY` (любая случайная строка), `ADMIN_LOGIN`/`ADMIN_PASSWORD` (логин/пароль для входа в `/admin`), `API_KEY` (любая строка — нужна, если будете дёргать `/search`/`/ingest` напрямую, не через админку). `YANDEX_API_KEY`/`YANDEX_FOLDER_ID`/`POLZA_API_KEY` — реальные ключи нужны, если хотите, чтобы загрузка документов и поиск реально работали (без них ingestion/search будут падать с ошибкой подключения к LLM/embedding API).

2. **Поднять Postgres и Qdrant:**
   ```bash
   docker compose up -d db qdrant
   ```

3. **Применить миграции и запустить сам сервис** (без Docker, напрямую из venv — проще для разработки и быстрее перезапускать):
   ```bash
   python -m venv venv
   venv\Scripts\activate        # Windows
   # source venv/bin/activate   # Linux/macOS
   pip install -r requirements-dev.txt
   alembic upgrade head
   hypercorn app.main:app --bind 0.0.0.0:8000 --reload
   ```
   В `.env` для такого запуска `POSTGRES_HOST`/`QDRANT_URL` должны указывать на `localhost` (а не на имена контейнеров `db`/`qdrant`) — именно так и заполнен `.env.example`, можно не менять.

4. Откройте в браузере `http://localhost:8000/admin/login` — дальше по инструкции ниже.

**Альтернатива — всё целиком в Docker** (без venv, ближе к production-запуску):
```bash
docker compose up -d --build
```
Подождите несколько секунд, пока сервис применит миграции и поднимется (`docker compose logs -f rag_service`), затем откройте `http://localhost:8000/admin/login`.

**Проверка, что всё поднялось:** `curl http://localhost:8000/api/v1/health` должен вернуть `{"status":"ok","database":"ok"}`.

## Вход

Откройте `/admin/login`, введите логин/пароль (выдаёт администратор сервиса, см. `ADMIN_LOGIN`/`ADMIN_PASSWORD` в `.env`). Сессия привязана к браузеру — повторно логиниться не нужно, пока не нажмёте «Выход» или не очистите куки.

## Добавление нового документа

1. Откройте **«Загрузка документа»** (`/admin/document-upload`).
2. Загрузите файл — `.pdf`, `.docx`, `.md` или `.txt`. Размер до 20 МБ, PDF — до 2000 страниц. **`.doc`/`.rtf` не поддерживаются** — сохраните такой файл в `.docx` (или `.pdf`) перед загрузкой.
3. Заполните поля:
   - **document_id** — короткий уникальный код документа, латиницей, например `fz-181-art21`. **Запомните его** — он понадобится для обновления и удаления этого документа в будущем.
   - **Категория** — выберите одну из пяти:
     - `labor_code` — ТК РФ;
     - `case_law` — судебная практика, разъяснения Пленумов ВС РФ;
     - `federal_law` — иные федеральные законы (не ТК РФ), например ФЗ-181;
     - `other_npa` — подзаконные акты (постановления Правительства и т.п.);
     - `authorial` — авторские статьи и систематизации.
   - **Название источника** — человекочитаемое, например «ФЗ-181, Статья 21».
   - **Аудитория** — `seeker` (соискатель), `employer` (работодатель) или `both` (оба).
   - **Тема** — короткий тег, например `quota`, `dismissal`, `workplace`.
   - **Версия** — дата редакции документа, например `2026-01-01`. Если документ новый — можно использовать дату загрузки.
   - **Дата вступления в силу** — когда эта редакция стала действующей.
4. Нажмите **«Загрузить и проиндексировать»**. Это запускает полную обработку (разбор → обогащение LLM → векторизация) — может занять от нескольких секунд до пары минут в зависимости от объёма документа. После успеха появится зелёное сообщение с числом проиндексированных фрагментов.

Если показалась красная ошибка — самые частые причины: не выбран файл, недопустимая категория, файл слишком большой, или временная недоступность LLM/embedding API (в этом случае просто попробуйте загрузить ещё раз).

## Обновление документа (новая редакция)

Если в документ внесли изменения (вышла новая редакция закона, поправили статью):

1. Загрузите файл так же, как новый документ (раздел выше), но укажите:
   - **тот же `document_id`**, что был у предыдущей версии;
   - **новую `version`** (например, была `2026-01-01`, стала `2026-03-01`);
   - новую `effective_date`, если она изменилась.
2. Остальные поля можно оставить как раньше или поправить при необходимости.

Сервис сам всё сделает правильно: сначала полностью загрузит и проиндексирует новую версию, и только после успеха удалит старую — пользователи Веры не останутся без ответа на время обновления. Старая версия документа после этого пропадёт из поиска, но останется видна в списке документов (см. ниже) для истории.

**Не нужно** отдельно удалять старую версию перед загрузкой новой — это сделает дубликаты или временный провал в покрытии. Просто загружайте новую версию с тем же `document_id`.

## Просмотр и удаление документов

Откройте **«Документ»** в меню слева (`/admin/document/list`) — список всех загруженных версий документов: `document_id`, версия, категория, активна ли сейчас.

- **Посмотреть, что реально проиндексировано** — кликните по `document_id` в списке. Откроется страница с текстом каждого фрагмента (чанка) и тем, какой заголовок и вопросы к нему сгенерировал LLM. Полезно, если результаты поиска выглядят странно — можно проверить, что документ разбился на чанки правильно.
- **Удалить** — кнопка удаления у строки списка убирает и саму запись, и все её фрагменты из поиска. Удаление одной строки = удаление именно этой версии документа (если у документа несколько версий — остальные не тронутся).

Удаление — необратимо (документ нужно будет загружать заново). Перед удалением убедитесь, что выбрали нужную строку.

## Тестирование поиска

Откройте **«Тестирование поиска»** (`/admin/search-test`) — позволяет проверить, что найдёт Вера на конкретный вопрос, без необходимости спрашивать через сам продукт.

1. Введите вопрос, как его мог бы задать пользователь (например, «Какая квота на трудоустройство инвалидов?»).
2. Опционально — сузьте фильтрами:
   - **Аудитория** — если хотите проверить поиск именно для соискателя или работодателя;
   - **Тема** — если ищете по конкретному тегу;
   - **Категория** — если хотите проверить поиск только внутри одной категории источников. Если оставить пустым — поиск идёт по всем пяти категориям сразу с гарантированным представительством каждой (так и работает поиск в продукте по умолчанию).
3. Нажмите «Искать».

Результат показывает:
- **Финальный результат** — то, что реально увидел бы пользователь Веры (после переранжирования LLM) — самое важное для проверки.
- **Dense / Sparse / RRF-склейка** — промежуточные списки кандидатов до финального отбора, для более глубокой диагностики (если итог выглядит неверным — здесь видно, на каком этапе потерялся нужный фрагмент).

Каждый тестовый запрос автоматически попадает в журнал поисковых запросов (как обычный поисковый запрос продукта) — это нормально, журнал нужен для анализа качества поиска.

## Дашборд — общее состояние сервиса

Откройте **«Дашборд»** (`/admin/dashboard`) для быстрой проверки:
- доступны ли Postgres и Qdrant;
- сколько документов и чанков сейчас в базе знаний;
- средняя скорость поиска за последние 24 часа;
- есть ли расхождения между реестром документов и реальным содержимым базы (если есть — будет явное предупреждение со списком, к админу/разработчику).

Полезно открыть после массовой загрузки документов — убедиться, что всё прошло штатно.

## Частые вопросы

**Можно ли загрузить документ с тем же `document_id`, не меняя `version`?** Можно — сервис просто перезапишет ту же версию (это безопасно, дублей не создаст), но новых данных, если файл не менялся, не появится. Если меняете содержимое — обязательно укажите новую `version`, иначе путаница в истории редакций.

**Что делать, если загрузка зависла или дала ошибку сети?** Просто повторите загрузку с теми же `document_id`/`version` — это безопасно, повторная отправка не создаст дублей.

**Слишком много попыток входа за раз?** Вход ограничен (5 попыток в минуту) для защиты от подбора пароля — подождите минуту и попробуйте снова.
