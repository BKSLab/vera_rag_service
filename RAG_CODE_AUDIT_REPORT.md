# Отчет по аудиту `vera_rag_service`

Дата аудита: 2026-07-04
Формат работы: первичный аудит выполнен по документации и исходному коду; после аудита закрыт первый критичный пункт по точечному обновлению статьи.

## Контекст аудита

Перед ревью кода были изучены все `.md` документы в корне проекта:

- `ADMIN_GUIDE.md`
- `FASTAPI_PATTERNS.md`
- `LLM_CLIENT_REFERENCE.md`
- `RAG_SERVICE_PLAN.md`
- `README.md`

Фокус аудита: юридический RAG-сервис, где наиболее важны корректность поиска информации в базе знаний, полнота retrieval, надежность обновления нормативных текстов, производительность hot path поиска и эксплуатационная устойчивость.

На этапе первичного аудита тесты не запускались: среда была в режиме `read-only`, а `pytest` обычно пишет `.pytest_cache` и временные артефакты. После реализации первого критичного пункта пройдены `ruff check` для измененных файлов, `pytest tests/integration/services/test_ingestion_service.py`, `pytest tests/api/endpoints/test_documents.py tests/unit/test_chunking.py`.

## Краткий вывод

Сервис в целом имеет зрелую архитектуру: слоистая структура, отдельные клиенты внешних API, Qdrant + Postgres, категорийно-сбалансированный hybrid retrieval, query expansion, LLM-reranker, журналирование поиска, админка и тесты.

Основные риски сейчас не в общей архитектуре, а в нескольких точках, которые напрямую влияют на юридическую корректность:

- точечное обновление статьи могло сделать новую редакцию невидимой для поиска - **закрыто реализацией и тестами**;
- гипотетические вопросы индексируются, но не участвуют в dense retrieval;
- реестр документов в Postgres не идемпотентен при повторной загрузке той же версии;
- схема существующей Qdrant-коллекции не валидируется на старте;
- очистка юридического текста не закрывает уже обнаруженный служебный шум корпуса;
- hot path LLM использует слишком длинные offline-таймауты.

## Приоритеты

| Приоритет | Что исправить | Почему важно |
|---|---|---|
| P0 | `ingest_section()` / точечная деактивация старых чанков | **Закрыто**: новая редакция остается актуальной, старые chunk IDs деактивируются адресно |
| P0 | Подключить `question_N` vectors к retrieval | Сейчас enrichment вопросами почти не повышает recall |
| P1 | Идемпотентный `documents(document_id, version)` | Админский реестр может расходиться с Qdrant |
| P1 | Валидация схемы Qdrant на старте | Сервис может стартовать на старой несовместимой коллекции |
| P1 | Очистка служебного шума юридических источников | Мусор попадает в эмбеддинги и ответы |
| P1 | Отдельные timeout/retry для hot path LLM | При деградации LLM поиск может ждать минуты |
| P2 | Golden set и замер recall/top-k | Нужна эмпирическая настройка retrieval |
| P2 | Улучшить sparse-нормализацию русского текста | Повысит точный поиск по юридическим формулировкам |

## Критичные проблемы

### 1. Точечное обновление статьи делает новую редакцию невидимой для поиска

Критичность: **Critical**
Статус: **закрыто реализацией и интеграционными тестами**

Файлы:

- `app/services/ingestion.py`
- `app/vectorstore/qdrant_client.py`
- `app/search/hybrid.py`

В `IngestionService.ingest_section()` новая редакция сначала записывается в Qdrant:

```python
await self.vector_store.upsert_chunks(embedded_chunks, document_metadata)
```

Затем вызывается:

```python
superseded = await self.vector_store.set_section_inactive(
    parent_id=parent_id,
    effective_until=request.effective_date,
)
```

`QdrantVectorStore.set_section_inactive()` выбирает все точки:

```python
FieldCondition(key='parent_id', match=MatchValue(value=parent_id))
FieldCondition(key='is_actual', match=MatchValue(value=True))
```

Проблема: только что добавленные новые чанки имеют тот же `parent_id` и `is_actual=True`, поэтому они тоже попадают под `set_payload({'is_actual': False, ...})`.

Поиск всегда добавляет фильтр:

```python
FieldCondition(key='is_actual', match=MatchValue(value=True))
```

Итог: после точечного обновления статьи новая актуальная редакция может быть помечена неактуальной и исчезнуть из выдачи.

Почему это критично для юридического RAG:

- пользователь может не получить действующую норму права;
- Agent Service увидит пустой или неполный retrieval;
- юридическая консультация будет построена без актуальной редакции статьи;
- дефект особенно опасен, потому что API вернет успешный `200`, а проблема проявится позже в поиске.

Выполненная реализация:

1. До upsert новой редакции получить список старых актуальных `chunk_id` по `parent_id`.
2. После успешного upsert помечать неактуальными только старые ID, а не весь `parent_id`.
3. В `QdrantVectorStore` вместо деактивации по всему `parent_id` добавлены операции получения старых актуальных point IDs и адресной деактивации только этих IDs.
4. Учтено, что число чанков статьи между редакциями может измениться:
   - было 1 чанк, стало 2 или больше;
   - было несколько чанков, стало 1;
   - было N чанков, стало M чанков с другими `chunk_number_in_section`;
   - текст изменился так, что часть deterministic `chunk_id` совпала, а часть стала новой.

   Поэтому корректный критерий устаревания - не `chunk_number_in_section` и не количество чанков, а именно набор старых актуальных point IDs, зафиксированный до записи новой редакции.
5. Добавлены интеграционные тесты на точечное обновление секции:
   - старая редакция деактивируется;
   - новая редакция остается `is_actual=True`;
   - покрыты сценарии изменения числа чанков `1 -> N` и `N -> 1`.

### 2. Гипотетические вопросы индексируются, но не участвуют в dense-поиске

Критичность: **High**

Файлы:

- `app/vectorstore/qdrant_client.py`
- `app/search/hybrid.py`
- `app/embeddings/embedder.py`
- `app/ingestion/enrichment.py`

Документация описывает LLM-enrichment как ключевой механизм повышения recall: для каждого чанка генерируются `hypothetical_questions`, которые должны помогать находить норму по бытовым вопросам пользователя.

Фактически:

- вопросы эмбеддятся;
- в Qdrant создаются named vectors `question_0..4`;
- но dense-поиск использует только:

```python
using=CHUNK_VECTOR_NAME
```

То есть `question_0..4` не запрашиваются ни в `dense_search()`, ни в `hybrid_search()`.

Почему это важно для юридического RAG:

- реальные пользователи спрашивают бытовым языком;
- юридическая норма часто сформулирована иначе;
- именно гипотетические вопросы должны закрывать этот разрыв;
- сейчас сервис платит latency и стоимость ingestion за вопросы, но не получает основной пользы в retrieval.

Предложение:

1. В `hybrid_search()` добавить dense lanes по `chunk` и каждому `question_N`.
2. Фьюзить эти lanes через RRF так же, как сейчас фьюзятся категории и sparse/dense.
3. Ограничить top-k на question-vector lanes отдельно, чтобы не раздувать candidate pool.
4. Добавить тест: чанк плохо совпадает по `chunk_vector`, но идеально совпадает по `question_0`; такой чанк должен попасть в `fused`.
5. Рассмотреть вариант отдельной точки на каждый гипотетический вопрос с `parent_chunk_id`, если named-vector RRF станет неудобен.

### 3. Повторная загрузка той же версии плодит дубли в Postgres-реестре

Критичность: **High**

Файлы:

- `app/db/models/document.py`
- `app/repositories/document.py`
- `app/db/alembic/versions/20260620_2200_add_documents_table.py`
- `ADMIN_GUIDE.md`

В `ADMIN_GUIDE.md` указано:

> Можно — сервис просто перезапишет ту же версию (это безопасно, дублей не создаст)

Qdrant действительно идемпотентен за счет deterministic `chunk_id`. Но таблица `documents` не имеет уникального ограничения на `(document_id, version)`, а `DocumentRepository.save_document()` всегда делает:

```python
self.db_session.add(document)
await self.db_session.commit()
```

Итог: повторная загрузка той же версии добавит новую строку в Postgres-реестр.

Почему это важно:

- админка покажет дубли документов;
- дашборд будет считать лишние записи;
- история версий станет недостоверной;
- оператор может удалить не ту строку или получить неверную картину состояния БЗ.

Предложение:

1. Добавить `UniqueConstraint('document_id', 'version')`.
2. Сделать upsert записи документа:
   - если версия уже есть, обновить `category`, `source_title`, `audience`, `topic`, `effective_date`, `is_active`;
   - если нет, вставить новую строку.
3. Миграцией дедуплицировать существующие строки, если они уже могли появиться.
4. Добавить интеграционный тест на повторный `ingest_document()` той же версии: одна строка в `documents`, те же chunk IDs в Qdrant.

## Высокий приоритет

### 4. `ensure_collection()` не проверяет схему существующей коллекции

Критичность: **High**

Файл: `app/vectorstore/qdrant_client.py`

Сейчас:

```python
exists = await self.client.collection_exists(self.collection_name)
if exists:
    return
```

Если коллекция уже существует, сервис не проверяет:

- размерность dense-векторов;
- наличие `chunk`, `question_0..4`;
- наличие sparse-вектора `bm25`;
- наличие payload-индексов;
- наличие `is_actual`, `parent_id`;
- настройки квантизации.

Почему это важно:

- проект уже менял размерность эмбеддингов и схему Qdrant;
- старая коллекция может остаться в volume;
- приложение стартует успешно, но поиск будет падать или работать не по ожидаемой схеме;
- оператор получит ошибку уже в runtime.

Предложение:

1. На старте читать `get_collection()`.
2. Сравнивать фактическую схему с ожидаемой.
3. При несовместимости падать fail-fast с понятным сообщением.
4. Для совместимых недостающих payload-indexes создавать их автоматически.
5. Для несовместимых vector/sparse изменений требовать явный reindex/migration.

### 5. Очистка юридического текста не закрывает уже найденный мусор корпуса

Критичность: **High**

Файлы:

- `app/ingestion/preprocess.py`
- `RAG_SERVICE_PLAN.md`

В `RAG_SERVICE_PLAN.md` после первой реальной загрузки ТК РФ зафиксированы проблемы:

- преамбулы ГАРАНТ;
- блоки «Информация об изменениях»;
- строки «см. текст статьи в предыдущей редакции»;
- утечка заголовков разделов/глав между статьями;
- фантомные секции из истории правок.

Но `clean_text()` сейчас делает только базовую очистку:

- `\r\n` -> `\n`;
- дефисные переносы;
- одиночные строки-номера страниц;
- схлопывание пустых строк.

Почему это важно:

- служебный шум попадает в `chunk.text`;
- по нему строятся dense и sparse embeddings;
- LLM-enrichment может сгенерировать вопросы про «где посмотреть предыдущую редакцию»;
- Agent Service может цитировать не норму, а артефакт справочной системы.

Предложение:

1. Добавить отдельный слой очистки правовых источников.
2. Начать с regex-правил под подтвержденные реальные артефакты.
3. Не удалять историю правок вслепую, если она может содержать юридически значимую дату; лучше отделять ее от текста нормы.
4. Добавить fixture-фрагменты из реального ТК РФ.
5. Проверять, что после очистки:
   - статья не теряет норму;
   - служебные строки не попадают в chunk;
   - `LAW_ARTICLE_PATTERN` не создает фантомные секции.

### 6. Hot path LLM использует offline-таймауты

Критичность: **High**

Файлы:

- `app/clients/llm.py`
- `app/dependencies/clients.py`
- `app/services/search.py`

`LlmClient` по умолчанию:

```python
DEFAULT_TIMEOUT_SECONDS = 90
DEFAULT_RETRIES = 3
```

Эти дефолты используются не только для offline enrichment, но и для hot path:

- query expansion;
- reranker.

Оба шага выполняются синхронно внутри `/search`.

Почему это важно:

- при деградации Polza один поисковый запрос может ждать очень долго;
- circuit breaker откроется только после нескольких полностью проваленных вызовов;
- SLA интерактивного поиска будет нестабильным;
- Agent Service может зависать на retrieval.

Предложение:

1. Разделить настройки:
   - enrichment: длинный timeout, больше retry;
   - query expansion: короткий timeout, 1-2 retry;
   - reranker: короткий timeout, 1-2 retry.
2. Добавить общий deadline на `SearchService.search_with_diagnostics()`.
3. Логировать `query_expansion_status` и `reranker_status`.
4. Для query expansion fallback к исходному запросу оставить.
5. Для reranker fallback сделать управляемым настройкой, см. следующую проблему.

## Средний приоритет

### 7. Fallback reranker'а может вернуть нерелевантные чанки

Критичность: **Medium/High**

Файл: `app/search/reranker.py`

Если LLM-reranker недоступен, код возвращает исходный RRF-порядок:

```python
return [chunk_id for chunk_id, _ in candidates[:top_n]]
```

Если LLM явно вернул `ranked_indices=[]`, сервис корректно возвращает пустой список. Но при техническом отказе LLM semantic gate фактически отключается.

Почему это важно:

- для юридического сервиса лучше явно сказать «не нашел подтвержденный источник», чем дать нерелевантный фрагмент;
- Agent Service может построить уверенный ответ из слабого RRF-кандидата;
- RRF-score не является абсолютной мерой релевантности.

Предложение:

1. Добавить в `SearchDiagnostics` и `search_logs` статус reranker'а:
   - `ok`;
   - `no_relevant`;
   - `fallback_unavailable`;
   - `fallback_invalid_output`.
2. В API опционально возвращать этот статус или хотя бы передавать его MCP/Agent Service.
3. Рассмотреть режим `strict_reranker=true`: при отказе reranker'а возвращать `chunks=[]`, а не RRF fallback.
4. Минимум: помечать fallback-результаты как низкодоверительные.

### 8. Top-K по категориям не валидирован на реальном корпусе

Критичность: **Medium**

Файлы:

- `app/core/settings.py`
- `app/search/hybrid.py`
- `RAG_SERVICE_PLAN.md`

Документация честно фиксирует, что `4+4` на категорию не валидировано на реальном корпусе.

Почему это важно:

- слишком маленький top-k потеряет важные источники;
- слишком большой top-k даст reranker'у шум;
- разные категории имеют разный объем и разную юридическую роль;
- `case_law` и `authorial` могут быть малочисленными, но критичными для полноты ответа.

Предложение:

1. Собрать golden set из 50-100 вопросов.
2. Для каждого вопроса вручную отметить обязательные источники/категории.
3. Мерить recall@k до reranker и после reranker.
4. Отдельно проверять:
   - вопросы, закрываемые только ТК РФ;
   - вопросы, где нужна судебная практика;
   - вопросы, где нужен ФЗ-181 или подзаконный акт;
   - вопросы по авторским систематизациям.
5. Настроить `dense_top_k_per_category` и `sparse_top_k_per_category` на данных, а не по предположению.

### 9. Sparse-поиск слишком простой для русского юридического текста

Критичность: **Medium**

Файл: `app/vectorstore/sparse.py`

Сейчас токенизация:

```python
_WORD_PATTERN = re.compile(r'\w+', re.UNICODE)
return _WORD_PATTERN.findall(text.lower())
```

Нет:

- нормализации `е/ё`;
- лемматизации;
- обработки сокращений;
- словаря юридических синонимов;
- нормализации номеров статей и процентов.

Почему это важно:

- русский язык сильно флективен;
- пользовательские формы слов могут не совпадать с формой в норме;
- юридические сокращения вроде `ТК`, `ТК РФ`, `ФЗ`, `ст.` требуют нормализации;
- sparse-поиск должен закрывать точные термины, но сейчас он делает это только на уровне простых токенов.

Предложение:

1. Добавить нормализацию `ё -> е`.
2. Добавить нормализацию юридических сокращений:
   - `ст.` -> `статья`;
   - `ТК РФ` -> `трудовой кодекс`;
   - `ФЗ` -> `федеральный закон`.
3. Рассмотреть легкую лемматизацию при индексации и запросе.
4. Добавить тесты на формы:
   - `инвалид`;
   - `инвалида`;
   - `инвалидов`;
   - `работодатель`;
   - `работодателя`.

### 10. Сверка Postgres/Qdrant слабая

Критичность: **Medium**

Файл: `app/admin/reconciliation.py`

Сверка проверяет только наличие версии документа в Qdrant:

```python
versions_in_qdrant = await vector_store.get_document_versions(document_id)
if version not in versions_in_qdrant:
    mismatches.append((document_id, version))
```

Она не проверяет, есть ли поисково-доступные чанки с `is_actual=True`.

Почему это важно:

- после бага в `ingest_section()` версия может существовать, но актуальных чанков не будет;
- дашборд покажет, что документ есть;
- поиск при этом не найдет норму.

Предложение:

1. Добавить проверку count по `document_id + version + is_actual=True`.
2. Отдельно показывать:
   - нет точек версии;
   - есть точки, но нет актуальных;
   - есть актуальные, но меньше ожидаемого минимума.
3. Для точечных обновлений сверять `parent_id`.

## Низкий приоритет

### 11. `PUT /document/{id}/sections/{section}` не мапит LLM/Embedding ошибки явно

Критичность: **Low/Medium**

Файл: `app/api/v1/endpoints/documents.py`

`update_section()` ловит только `ValueError`. В отличие от `/ingest`, ошибки `LlmApiRequestError` и `EmbeddingApiRequestError` не мапятся явно в контролируемый ответ.

Предложение: привести обработку ошибок к тому же контракту, что у `/ingest`.

### 12. Reranker получает мало юридического контекста

Критичность: **Low/Medium**

Файлы:

- `app/services/search.py`
- `app/search/reranker.py`

Reranker получает только текст чанка. Для юридической релевантности могут быть важны:

- `source_title`;
- `category`;
- `section_number`;
- `section_title`;
- `effective_date`;
- `version`.

Предложение: включать в prompt компактный metadata header кандидата. Особенно полезно для различения ТК РФ, ФЗ-181, Пленума и авторского комментария.

### 13. Нет полноценной retention-политики для `search_logs`

Критичность: **Low/Medium**

Файл: `app/db/models/search_log.py`

`search_logs` хранит пользовательский запрос, который может содержать сведения об инвалидности и ситуации пользователя. Дашборд считает latency за 24 часа, но сама таблица не очищается и не партиционируется.

Предложение:

1. Определить retention срок.
2. Добавить регламент очистки или партиционирование.
3. Рассмотреть маскирование/обезличивание старых запросов.

## Итог

Проект уже близок к рабочему production-grade RAG-сервису, но для юридического домена нужно закрыть несколько точек, где текущая реализация может дать не просто технический сбой, а юридически неверный результат.
