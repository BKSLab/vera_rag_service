# FastAPI — эталонные паттерны архитектуры

> Обобщённый набор архитектурных решений и стиля кода для проектов на FastAPI.
> Не привязан к конкретной бизнес-логике — используется как референс при старте
> и разработке любого нового проекта. Все примеры построены на абстрактной
> сущности `Item` — при переносе в конкретный проект имена меняются на доменные,
> сама структура и набор приёмов остаются неизменными.
> Язык кода, докстрингов, логов и комментариев — русский.

---

## 1. Слоистая архитектура

Строгая однонаправленная цепочка зависимостей:

```
Endpoint → Service → Repository → DB
                 ↘ Client (внешний API / LLM)
```

Правила:
- Эндпоинт не знает о репозитории и о SQLAlchemy. Он знает только сервис.
- Сервис не знает о HTTP (не оперирует `HTTPException`, `status_code` и т.п.) и не пишет SQL. Он оперирует репозиториями, клиентами и доменными исключениями.
- Репозиторий не знает о бизнес-логике. Его методы — это операции с одной таблицей/моделью: `get_*`, `save_*`, `delete_*`, `get_count_*`. Никакой валидации, никаких пересчётов — это работа сервиса.
- Каждый слой бросает **свои** исключения (см. раздел 8). Слой выше перехватывает исключения только нижних слоёв, никогда не "чужие".
- Между сервисами допустима горизонтальная зависимость (один сервис вызывает другой, например `ItemsService` зависит от `OwnersService`), но это всегда явная зависимость через конструктор, а не импорт внутри метода.

## 2. Структура проекта

```
app/
├── main.py                 # Точка входа: FastAPI(), lifespan, роутеры, exception handlers
├── core/
│   ├── settings.py          # Pydantic Settings, разбитые по доменам (App/DB/Llm/...)
│   └── config_logger.py     # Инициализация логирования из .ini, экспорт root-логгера проекта
├── api/
│   └── v1/
│       └── endpoints/        # Один файл = один роутер на одну сущность
├── dependencies/
│   ├── db_session.py         # Depends на сессию БД
│   ├── repositories.py       # Depends-фабрики репозиториев
│   ├── services.py           # Depends-фабрики сервисов
│   ├── clients.py             # Depends-фабрики HTTP/LLM клиентов
│   └── http_client.py         # Depends на httpx.AsyncClient (с lifespan-управлением)
├── services/                  # Бизнес-логика, один файл = одна сущность/процесс
├── repositories/               # Доступ к БД, один файл = одна модель
├── clients/                     # Обёртки над внешними API/LLM
├── db/
│   ├── session.py               # engine + async_sessionmaker
│   ├── models/                   # SQLAlchemy ORM-модели
│   └── alembic/                   # Миграции
├── schemas/                       # Pydantic-схемы запросов/ответов
├── exceptions/                     # Кастомные исключения, по слоям/доменам
├── admin/                           # sqladmin: auth.py + views.py + __init__.py (create_admin)
├── background_tasks/                 # Периодические задачи (очистка, синхронизация)
└── utils/                             # Точечные утилиты без состояния (check_db, security)
```

Принцип: **один файл — одна сущность**, во всех слоях одновременно. Если в проекте есть сущность `Item`, у неё есть `db/models/items.py`, `repositories/items.py`, `schemas/items.py`, и обычно часть `services/items.py`, `api/v1/endpoints/items.py`, `exceptions/items.py`. Это позволяет агенту (и человеку) находить весь контекст по сущности без grep по всему проекту.

## 3. Конфигурация (`core/settings.py`)

- `pydantic_settings.BaseSettings`, разбитая на несколько логических классов (`AppSettings`, `DBSettings`, `LlmSettings`, ...), а не один плоский класс. Каждый домен настроек — отдельный класс с собственным `SettingsConfigDict`.
- Общий класс-агрегатор `Settings` собирает домены через `Field(default_factory=...)`.
- `SecretStr` — только для того, что действительно секрет (пароль, токен, API-ключ). Хост, порт, имя пользователя и имя БД — это конфигурация, не секрет: их маскировка в логах не даёт реальной защиты, а в диагностических сообщениях ("не удалось подключиться к хосту X") они почти всегда нужны в открытом виде. Оборачивать в `SecretStr` всё подряд "на всякий случай" — ложная безопасность и лишний `.get_secret_value()` в каждом месте использования.
- Вычисляемые значения (например, строка подключения к БД) — `@property` на классе настроек, а не сборка строки в месте использования.
- Доступ к настройкам — только через `get_settings()`, обёрнутую в `@lru_cache`. Никогда не создавать `Settings()` напрямую где-либо кроме самого `get_settings`.

```python
class DBSettings(SettingsBase):
    """Настройки подключения к БД."""
    postgres_host: str
    postgres_port: int
    postgres_user: str
    postgres_password: SecretStr
    postgres_name: str

    @property
    def url_connect(self) -> str:
        return (
            f'postgresql+asyncpg://{self.postgres_user}:'
            f'{self.postgres_password.get_secret_value()}@'
            f'{self.postgres_host}:{self.postgres_port}/'
            f'{self.postgres_name}'
        )

@lru_cache
def get_settings() -> Settings:
    return Settings()
```

### Минимальный `.env` для БД

Без подключения к БД приложение не запустится (см. раздел 5 — проверка БД в lifespan), поэтому это первое, что должно быть в `.env` любого нового проекта:

```dotenv
# --- База данных (Postgres) ---
POSTGRES_HOST=localhost
POSTGRES_PORT=5432
POSTGRES_NAME=api_wfo
POSTGRES_USER=postgres
POSTGRES_PASSWORD=123_321_asdf
```

- Имена переменных совпадают с именами полей `DBSettings` в верхнем регистре (`postgres_host` → `POSTGRES_HOST`) — `pydantic-settings` подхватывает их автоматически по этому правилу, без `env_prefix` или другой настройки маппинга. Префикс `POSTGRES_` зашит прямо в имя поля, а не выводится через конфиг класса — так имя переменной видно сразу при чтении `DBSettings`, без необходимости смотреть `SettingsConfigDict`.
- Значения в `.env` не нужно оборачивать в кавычки (`POSTGRES_HOST=localhost`, не `POSTGRES_HOST='localhost'`) — кавычки попадут в значение как часть строки у некоторых парсеров `.env`, если не пользоваться именно `python-dotenv`/`pydantic-settings`, которые их аккуратно срезают. Безопаснее не привыкать к кавычкам по умолчанию и добавлять их только когда в значении реально есть пробелы.
- `POSTGRES_PORT` — обязательное поле без значения по умолчанию в `DBSettings`, хотя `5432` почти всегда стандартный порт. Это сделано намеренно: стандартный порт внутри контейнера и порт, который реально проброшен наружу в `docker-compose.yml`, на локальной машине часто отличаются — явное требование значения в `.env` страхует от тихого подключения не туда, куда ожидалось.
- `POSTGRES_HOST` указывается **без порта внутри строки** (`localhost`, не `localhost:5432`) — порт всегда отдельное поле `postgres_port`, не часть хоста. Иначе при сборке `url_connect` порт задвоится или потеряется.
- `.env.example` в репозитории должен содержать те же пять переменных с плейсхолдер-значениями (`POSTGRES_PASSWORD=changeme` и т.п.) — это уже зафиксировано как общее правило в разделе 16.

## 4. Логирование

- Конфигурация через `logging.ini` (`logging.config.fileConfig`), не через `dictConfig` в коде — конфиг логирования живёт отдельно от кода и его легко поменять без деплоя нового образа.
- Один корневой логгер проекта с уникальным именем (например, по имени пакета приложения), `propagate=0`, чтобы не дублировать вывод через root.
- В каждом модуле — собственный `logger = logging.getLogger(__name__)`, а не импорт общего логгера. Так в логах видно, из какого файла пришла запись, и легко настроить уровень для конкретного модуля.
- Формат лога включает `funcName`, чтобы сразу видеть, в какой функции произошло событие:
  `%(asctime)s - %(name)s - %(levelname)s - %(funcName)s - %(message)s`
- **Эмодзи-префиксы** как визуальный маркер типа события — это сознательный выбор для быстрого визуального сканирования логов глазами и грепа по эмодзи:
  - 🚀 — начало обработки запроса/процесса
  - ✅ — успешное завершение
  - ❌ — ошибка
  - ⚠️ — предупреждение, не критично, обработано
  - 🔍 — поиск / запрос к внешнему источнику
  - 🔄 — обновление/повтор операции
  - ➕ / 🗑️ — добавление / удаление
  - 📋 — получение списка
  - 💾 — операция с БД (сохранение)
  - ⚡ — параллельное/конкурентное выполнение
  - 🤖 — операция, связанная с LLM/AI
- Логирование с подстановками `%s`, **не f-строками** — `logger.info("...%s...", value)`, а не `logger.info(f"...{value}...")`. Причина: подстановка не выполняется, если уровень лога отфильтрован — экономия на ненужном форматировании, плюс корректная работа с `pprint`/большими структурами.
- Большие структуры (словари, списки) логируются через `pprint.pformat(...)`, а не напрямую — иначе лог превращается в нечитаемую простыню.
- `logger.exception(...)` (с traceback) — только в точке окончательной обработки ошибки (как правило, в эндпоинте перед `raise HTTPException`). Внутри сервисов/репозиториев — `logger.error(..., exc_info=True)` или `logger.warning(...)`, если ошибка некритична и обработана (есть fallback).
- На входе и выходе из каждого публичного метода эндпоинта — пара `logger.info` (🚀 в начале, ✅ в конце), с подстановкой ключевых параметров запроса.

## 5. Точка входа (`main.py`) и lifespan

- Инициализация тяжёлых/обязательных для старта ресурсов — через `@asynccontextmanager lifespan`, а не через `@app.on_event` (устаревший API).
- В lifespan — обязательная проверка доступности критичных внешних зависимостей (БД) при старте, с понятным логированием (🚀 → ✅) и явным падением приложения (`raise`), если проверка не прошла. Приложение не должно подниматься в "сломанном" виде.
- Прогрев/предзагрузка справочных данных выполняется тоже в lifespan, до `yield`, по идемпотентному паттерну: проверить, есть ли уже корректные данные в БД (включая проверку **ожидаемого количества записей**, а не только "не пусто") → если нет, прочитать из локального JSON-файла-фикстуры → сохранить → **повторно проверить**, что данные действительно появились → если нет, поднять исключение загрузки. Повторный запуск приложения на уже заполненной БД не делает лишней работы и не дублирует записи.
- Несколько шагов предзагрузки (например, справочник A и связанный с ним справочник B), даже логически независимые, выполняются **последовательно** (`await` один за другим), если они идут через одну и ту же `AsyncSession` — `asyncio.gather` на shared-сессии создаёт состояние гонки внутри SQLAlchemy и приводит к трудноуловимым ошибкам. Конкурентность безопасна только когда у каждой задачи своя сессия/клиент.
- `app.mount(...)` для статики и подключение админки (`create_admin(app, engine)`) — сразу после создания `app`, до объявления handlers и роутеров.
- Глобальные exception handlers (`@app.exception_handler(...)`) — для типовых "системных" ошибок (`RequestValidationError`, `RateLimitExceeded`), не для доменных. Доменные исключения обрабатываются локально в каждом эндпоинте, потому что только эндпоинт знает, какой HTTP-статус и комментарий уместен для конкретного клиента API.
- Роутеры подключаются через `app.include_router(router, prefix=...)`, версионирование API закладывается с первого дня (`/api/v1`).

## 6. Dependency Injection

Трёхуровневая система Depends-фабрик, зеркалящая слои архитектуры, плюс эндпоинт как потребитель верхнего уровня:

- **Уровень 0** — `dependencies/db_session.py`: низкоуровневая зависимость сессии БД (`AsyncGenerator` + `async with`).
- **Уровень 1** — `dependencies/repositories.py`: фабрики репозиториев, принимающие сессию через `Depends`.
- **Уровень 2** — `dependencies/services.py`: фабрики сервисов, принимающие репозитории/клиенты/другие сервисы через `Depends`.
- **Уровень 3** — эндпоинт: не фабрика, а потребитель — получает готовый объект сервиса одним параметром и ничего не знает об уровнях 0–2.

Каждая фабрика сопровождается готовым типом-алиасом `Annotated[Тип, Depends(фабрика)]` с суффиксом `Dep`, который и используется в сигнатурах эндпоинтов и других фабрик. Это убирает повторение `Depends(...)` по всему коду и даёт единую точку, где можно изменить способ создания объекта.

Полная цепочка для одной сущности, от сессии БД до эндпоинта — четыре файла:

```python
# dependencies/db_session.py — уровень 0: сессия БД
async def get_db_session() -> AsyncGenerator[AsyncSession, None]:
    async with async_session_factory() as db_session:
        yield db_session

DbSessionDep = Annotated[AsyncSession, Depends(get_db_session)]
```

```python
# dependencies/repositories.py — уровень 1: репозиторий поверх сессии
def get_items_repository(session: DbSessionDep) -> ItemsRepository:
    return ItemsRepository(session)

ItemsRepositoryDep = Annotated[
    ItemsRepository, Depends(get_items_repository)
]
```

```python
# dependencies/services.py — уровень 2: сервис поверх репозитория (и любых других зависимостей)
async def get_items_service(
    items_repository: ItemsRepositoryDep,
    other_service: OtherServiceDep,
) -> ItemsService:
    """Фабрика для создания экземпляра сервиса работы с сущностью Item."""
    return ItemsService(
        items_repository=items_repository,
        other_service=other_service,
    )

ItemsServiceDep = Annotated[ItemsService, Depends(get_items_service)]
```

```python
# api/v1/endpoints/items.py — уровень 3: эндпоинт получает готовый сервис одним параметром
@router.get(path="/{item_id}", response_model=ItemSchema)
async def get_item(item_id: str, service: ItemsServiceDep) -> ItemSchema:
    return await service.get_item(item_id=item_id)
```

Эндпоинт не видит ни `db_session`, ни `ItemsRepository` — FastAPI разворачивает граф зависимостей сам, а в сигнатуре остаётся только осмысленное имя `service`. Каждый уровень знает только о том, что стоит непосредственно под ним.

### Авторизация как Depends-зависимость

Проверка API-ключа реализуется как обычная Depends-зависимость (`VerifyApiKeyDep`), которая сама перехватывает доменные исключения и поднимает `HTTPException` — это единственное место, где Depends-слой сам бросает HTTP-исключение напрямую, потому что у него нет "вышестоящего" эндпоинта, который сделает это за него.

**Двухуровневая авторизация.** Если в проекте есть и обычные клиенты API, и привилегированные операции управления самим API (выпуск/отзыв ключей), это две независимые плоскости доступа с разными секретами:
- Обычный доступ — `X-API-Key` в заголовке, хешированный ключ в БД, проверяется через сервис и репозиторий (генерация и проверка ключа — раздел 9, подраздел «Секреты и API-ключи»).
- Управление ключами — отдельный `X-Master-Key`, сверяется напрямую с одним значением из `Settings` (`master_api_key.get_secret_value()`), без обращения к БД. Не подменять одно другим: мастер-ключ не должен попадать в таблицу ключей, а обычный API-ключ не должен давать доступ к управлению ключами.

### Жизненный цикл HTTP-клиента для внешних API

`httpx.AsyncClient`, который передаётся в клиентов внешних API (раздел 13) через `Depends`, может жить по одной из двух схем — выбор фиксируется один раз и одинаково для всех клиентов в проекте, не смешивается без причины:
- **per-request** (по умолчанию, если нет причин для другого) — `async def get_http_session(): async with httpx.AsyncClient(...) as client: yield client`, новый клиент на каждый запрос. Проще, безопаснее в плане утечек состояния между запросами, не требует синхронизации жизненного цикла с `lifespan`.
- **shared/lifespan-managed** — один `httpx.AsyncClient` создаётся в `lifespan` и переиспользуется во всех запросах через `app.state`. Снижает накладные расходы на установку соединений при высокой частоте запросов к одному и тому же внешнему хосту (connection pooling между запросами), но требует аккуратного закрытия при остановке приложения. Оправдан, когда профилирование показывает, что пересоздание соединений — реальное узкое место.

## 7. Эндпоинты (`api/v1/endpoints/*.py`)

Каждый эндпоинт оформляется максимально полно для автогенерируемой документации:

```python
@router.post(
    path="/items",
    status_code=status.HTTP_201_CREATED,
    summary="Краткое описание для списка эндпоинтов",
    description="Подробное описание бизнес-смысла операции.",
    operation_id="createItem",
    response_description="Что именно возвращается в ответе",
    responses={
        201: {"description": "...", "content": {"application/json": {"example": {...}}}},
        404: {"description": "...", "content": {"application/json": {"example": {"detail": "..."}}}},
        500: {"description": "...", "content": {"application/json": {"example": {"detail": "..."}}}},
    },
    response_model=ResponseSchema,
)
async def endpoint_name(
    data: RequestSchema,
    service: ServiceDep,
) -> ResponseSchema:
    """Что делает эндпоинт.

    Args:
        data: ...
        service: ...

    Returns:
        ...
    """
    logger.info("🚀 Запрос POST /items. Параметр: %s.", data.field)
    try:
        result = await service.do_something(...)
        logger.info("✅ Запрос POST /items выполнен.")
        return result
    except (DomainErrorA, DomainErrorB, DomainErrorC) as error:
        logger.exception("❌ Ошибка при ... Параметр: %s. Детали: %s", data.field, error)
        raise HTTPException(status_code=error.status_code, detail=error.detail)
```

Обязательные элементы:
- `summary`, `description`, `response_description`, `operation_id` — заполняются всегда, даже для тривиальных эндпоинтов. `operation_id` в camelCase — он используется при генерации клиентских SDK.
- `responses` со всеми реалистичными кодами ошибок (401/403/404/422/500 — в зависимости от того, что реально может произойти), с примером тела ответа. Это не формальность — именно отсюда фронтенд и клиенты API узнают контракт ошибок без чтения кода.
- Параметры запроса — всегда `Annotated[Тип, Query(description=...)]` / `Path(description=...)`, никогда "голые" аргументы без описания.
- Docstring в Google-стиле (`Args`/`Returns`/`Raises`) — обязателен даже когда сигнатура самодокументирована, потому что докстринг описывает поведение, а не сигнатуру.
- `try/except` перехватывает **только перечисленные** доменные исключения нижних слоёв (никогда голый `except Exception`). Если исключение не предусмотрено — пусть всплывёт как 500 со стектрейсом, это сигнал, что в коде пропущен кейс.
- Конвертация доменного исключения в HTTP-ответ — всегда одинаковая идиома: `raise HTTPException(status_code=error.status_code, detail=error.detail)`, потому что у каждого доменного исключения есть `status_code` и `.detail` (см. раздел 8).

## 8. Исключения — трёхслойная система

Каждый архитектурный слой имеет собственную иерархию исключений в `exceptions/`, разбитую по доменам, а не по слоям одним файлом:

```python
class ItemsRepositoryError(Exception):
    """Исключение для класса репозитория для работы с записями Item."""
    status_code = status.HTTP_500_INTERNAL_SERVER_ERROR

    def __init__(self, error_details: str):
        self.error_details = error_details
        super().__init__(self.error_details)

    def __str__(self) -> str:
        return f"Ошибка в ItemsRepository. Подробности: {self.error_details}"

    @property
    def detail(self) -> str:
        return f"Ошибка базы данных при обработке записей Item. Подробности: {self.error_details}"
```

Правила:
- У каждого исключения, которое может всплыть до эндпоинта, есть классовый атрибут `status_code` (HTTP-код, который из него следует сделать) и свойство `detail` (текст для клиента API — без внутренних деталей реализации, но с достаточным контекстом).
- `__str__` — для внутреннего лога (может включать больше технических деталей), `.detail` — для внешнего ответа (без утечки внутренностей).
- Исключения "не найдено" (`ItemNotFoundError`, `ItemsNotFoundError`) — отдельный класс, а не общая `ServiceError` с кодом 404, потому что у каждого такого исключения свой набор атрибутов для логирования (`item_id`, `owner_id`, `external_source`...).
- Исключение более низкого слоя **не пробрасывается дальше как есть** — оно либо перехватывается на границе и оборачивается в исключение текущего слоя (`raise ServiceError(...) from error`), либо явно перечислено как допустимое для прохождения сквозь слой (если у него уже подходящий контракт).
- Репозиторий перехватывает `SQLAlchemyError` (и в общем случае `Exception`) вокруг каждого DB-вызова, делает `rollback()` сессии при ошибке и поднимает свой `*RepositoryError` — вызывающий код никогда не видит `SQLAlchemyError` напрямую.
- Нарушение уникального ограничения — это не просто "ошибка БД", а конкретная доменная ситуация ("уже существует"). Такой случай перехватывается **отдельным** `except IntegrityError` (более узкий тип, идёт раньше общего `except SQLAlchemyError`) и оборачивается в специфичное исключение (`ItemAlreadyExistsError`, не общий `*RepositoryError`), чтобы сервис/эндпоинт могли вернуть осмысленный 409, а не общий 500.
- `detail` не обязан быть property — если у исключения нет данных, зависящих от контекста конкретного вызова (например, ошибка совпадения мастер-ключа всегда формулируется одинаково), `detail` объявляется как обычный **атрибут класса** (`detail = "Неверный мастер-ключ."`). `@property` используют только когда текст реально собирается из атрибутов конкретного экземпляра (id, префикс, локация и т.п.).
- У части "внутренних" исключений (которые перехватываются и retry'ятся в пределах одного клиента и никогда не пересекают границу слоя — например, ошибка одной попытки запроса к LLM до того, как исчерпаны все retry) намеренно нет `status_code`/`detail`. Эти атрибуты появляются только у финального исключения, которое действительно может всплыть до сервиса/эндпоинта.

## 9. Сервисный слой

- Класс сервиса принимает в конструкторе все свои зависимости (репозитории, клиенты, другие сервисы) — никаких глобальных импортов репозиториев внутри методов.
- Константы поведения (TTL кэша, лимиты конкурентности, пороги валидации) — `UPPER_CASE` атрибуты класса сервиса, а не магические числа в коде метода. Это делает поведение настраиваемым и явным при чтении класса.
- Публичные методы сервиса — это use-case'ы ("получить список записей с пагинацией", "сформировать отчёт по сущности"), а не CRUD-обёртки над репозиторием. Подготовка данных, ветвление по бизнес-правилам, агрегация из нескольких источников — здесь.
- Внутренняя декомпозиция — через `_приватные` методы, сгруппированные комментарием-разделителем по смысловым блокам (например, `# Блок методов экспорта`, `# Блок приватных методов для валидации...`). Это держит класс читаемым даже когда в нём 20+ методов.
- Конкурентные операции к нескольким независимым источникам — `asyncio.gather(..., return_exceptions=True)`, с последующей точечной обработкой каждого результата (`isinstance(result, ИзвестноеИсключение)` → ветка деградации, не падение всего запроса из-за одного недоступного источника).
- Ограничение конкурентности там, где это нужно (защита от перегрузки внешнего API/БД при batched-операциях) — `asyncio.Semaphore`, хранится как атрибут сервиса (`self.semaphore`), лимит — константа класса.
- CPU-bound операции (парсинг, трансформация больших структур) внутри async-кода — через `loop.run_in_executor(None, functools.partial(...))`, чтобы не блокировать event loop.
- Паттерн "TTL-кэш в своей же таблице БД": перед обращением к внешнему источнику сервис проверяет `updated_at`/время последнего успешного запроса по доменному ключу; если данные младше TTL и без признаков ошибки в последней попытке — отдаёт их из БД без похода во внешний API. Иначе выполняет полный цикл сбора и обновляет отметку времени. Это типовой способ резко снизить нагрузку на нестабильные внешние API без отдельного кэширующего слоя (Redis и т.п.), когда домен это позволяет.
- Деградация при частичном отказе: если один из нескольких источников данных недоступен, сервис логирует это (`⚠️`/`❌`) и продолжает с тем, что получилось, поднимая исключение только если отказали **все** источники одновременно.
- Валидация и нормализация входных доменных значений (не относящаяся к Pydantic-схемам запроса) — приватными методами сервиса с явными шагами (`_is_*_exist`, `_validate_*`, `_normalize_*`), а не одним монолитным регэкспом.

### Секреты и API-ключи (генерация и хранение)

- Пароли и пользовательские секреты — хешируются (`passlib.CryptContext` с `bcrypt`), сравнение — только через `verify_password`, никогда вручную через `==`.
- API-ключи для клиентов сервиса генерируются по схеме "префикс + случайный секрет" (`secrets.token_urlsafe`, не `random`/`uuid4` — криптографически стойкий генератор): `{prefix}_{secret}` отдаётся клиенту один раз и больше не восстановим; в БД хранится **хеш** полного ключа плюс отдельный короткий **db_prefix** (первые N символов секрета) для быстрого поиска по индексу без полного перебора и сравнения хешей со всей таблицей.
- При проверке входящего ключа: сначала вычисляется `db_prefix` от присланного значения → находится кандидат в БД по этому префиксу (`O(1)` по индексу) → и только потом делается дорогая операция `verify_password` именно для этого одного кандидата. Так бизнес-логика не сравнивает хеш входящего ключа со всеми ключами в таблице.
- Полный ключ никогда не сохраняется и не логируется — в логах и в админке фигурирует только `db_prefix`/`api_key_prefix`.

### Нормализация данных из внешних источников (сервис парсинга)

Когда у проекта несколько внешних источников с разной/грязной структурой данных, которые должны лечь в единую внутреннюю модель, выделяется отдельный сервис парсинга (не часть основного бизнес-сервиса):
- Все "дефолтные" значения для отсутствующих полей — поименованные константы класса (`DEFAULT_NOT_SPECIFIED`, `DEFAULT_CONTACT`, `DEFAULT_CATEGORY`...), а не разбросанные по коду строки-литералы — единая точка, если потребуется поменять текст-заглушку.
- Извлечение вложенных опциональных полей из "грязного" внешнего JSON — всегда защищённое: `record.get("key", {}) or {}` перед следующим `.get(...)`, никогда не предполагается, что промежуточный узел гарантированно существует и имеет ожидаемый тип.
- Метод парсинга одной записи оборачивается в `try/except Exception` целиком, с финальным `raise СвоёParseError(...)`, включающим достаточно контекста (id записи, источник) для диагностики конкретной сломанной записи без падения всего пакета.
- Парсинг списка записей не прерывается из-за одной "плохой" записи молча — либо это пропуск с логированием, либо явное исключение, в зависимости от того, является ли элемент критичным для бизнес-смысла результата.
- Финальная очистка значений перед сохранением в БД — отдельный шаг `_sanitize_*` (например, удаление `\x00`/неразрывных пробелов, которые ломают запись в PostgreSQL), применяется к каждой записи в одном месте, а не точечными `.replace()` по всему парсеру.
- Постобработка длинного неструктурированного текста (HTML-описания и т.п.) — отдельные переиспользуемые приватные методы (очистка тегов регэкспом, разбиение на абзацы по границам предложений с ограничением длины), используемые во всех методах парсинга, а не продублированные в каждом.

## 10. Репозитории

- Один репозиторий = одна модель/таблица (изредка — тесно связанная пара таблиц).
- Конструктор принимает `db_session: AsyncSession`, хранит как `self.db_session`.
- Каждый публичный метод — это один SQL-запрос (через SQLAlchemy Core/ORM `select`/`insert`/`update`/`delete`), оборачивается в `try/except (SQLAlchemyError, Exception)`, при ошибке — `rollback()` и `raise СвоёRepositoryError(...) from error`.
- Чтение не делает `commit()`. Запись — обязательный `commit()` после `execute()`, и обязательный `rollback()` в `except`.
- Массовая вставка — батчами (`SAVE_BATCH_SIZE` как константа класса), с логированием прогресса по батчам и индивидуальной обработкой ошибки конкретного батча, чтобы в логе было видно, какой именно батч сломался.
- Фильтрация — собирается через последовательные `stmt = stmt.where(...)` по присутствующим опциональным параметрам, а не через построение строки SQL.
- Методы подсчёта (`get_count_*`) — отдельные от методов получения данных (`get_*`), даже если фильтры идентичны, чтобы можно было получить total для пагинации без выгрузки самих строк.
- Аномалии данных, которые не являются ошибкой, но достойны внимания (например, найдено более одной записи там, где ожидалась одна) — логируются `logger.warning`, не превращаются в исключение, если есть разумный способ продолжить (взять первую запись).

## 11. Модели SQLAlchemy

```python
from datetime import datetime
from typing import Optional

from sqlalchemy import DateTime, ForeignKey, String, Text, UniqueConstraint, func
from sqlalchemy.orm import Mapped, mapped_column

from .base import Base


class Item(Base):
    """Модель для хранения сущности Item."""

    __tablename__ = 'items'

    __table_args__ = (
        UniqueConstraint('external_id', 'owner_id', name='unique_items_external_id__owner_id'),
    )

    id: Mapped[int] = mapped_column(
        primary_key=True,
        comment='Уникальный идентификатор записи.',
    )
    external_id: Mapped[str] = mapped_column(
        String(length=300),
        nullable=False,
        doc='Идентификатор сущности во внешней системе.',
        comment='ID записи на стороне источника данных.',
    )
    owner_id: Mapped[int] = mapped_column(
        ForeignKey('owners.id', ondelete='CASCADE'),
        nullable=False,
        doc='Владелец записи.',
        comment='Внешний ключ на таблицу owners. При удалении владельца запись удаляется.',
    )
    title: Mapped[str] = mapped_column(
        Text,
        nullable=False,
        doc='Название сущности.',
        comment='Человекочитаемое название записи.',
    )
    description: Mapped[Optional[str]] = mapped_column(
        Text,
        nullable=True,
        doc='Описание сущности.',
        comment='Произвольное текстовое описание, может отсутствовать.',
    )
    updated_at: Mapped[datetime] = mapped_column(
        DateTime(timezone=True),
        server_default=func.now(),
        onupdate=func.now(),
        nullable=False,
        doc='Дата последнего обновления записи.',
        comment='Метка времени последнего изменения записи, обновляется автоматически на стороне БД.',
    )

    def __repr__(self) -> str:
        return f"<Item(id={self.id}, external_id='{self.external_id}', title='{self.title}')>"
```

Ключевые моменты этого примера:
- **Каждое поле документировано дважды** — `doc=` для разработчика/IDE, `comment=` для самой БД (виден в любом SQL-клиенте независимо от кода).
- `__table_args__` с `UniqueConstraint` и явным `name=`, когда уникальность составная (а не на одном поле).
- `ForeignKey(..., ondelete='CASCADE')` — каскад описан на уровне схемы БД, не оставлен только на совести кода приложения.
- `onupdate=func.now()` для `updated_at` — обновление метки времени на стороне БД при каждом `UPDATE`, без ручного выставления в Python-коде.
- `__repr__` — id плюс 1–2 ключевых поля, достаточно для логов и отладки, не весь объект целиком.

Дальше — наследование от единой `Base(DeclarativeBase)` (плюс промежуточные абстрактные модели для общих наборов полей у родственных таблиц, например общая базовая модель для записей, поступающих из разных внешних источников с одинаковой структурой).
- Имя класса — PascalCase в единственном или множественном числе по смыслу сущности; `__tablename__` — snake_case, множественное число.
- Документирование **каждого** поля двумя независимыми способами:
  - `doc=` — для разработчика, видно через `__doc__`/IDE/автогенерацию.
  - `comment=` — для самой БД, видно в любом SQL-клиенте (DBeaver, psql `\d+`), не зависит от того, открыт ли код проекта.
- У каждой модели — `__repr__`, возвращающий компактное и информативное представление (id + 1-2 ключевых поля), для удобной отладки и логов.
- Уникальность на уровне домена — через `UniqueConstraint` с явным `name=`, а не только через `unique=True` на одном поле, если уникальность составная.
- Внешние ключи — с явным `ondelete=` (как правило `CASCADE` для зависимых сущностей), чтобы целостность данных не держалась только на коде приложения.
- Поля времени (`created_at`/`updated_at`) — `DateTime(timezone=True)`, `server_default=func.now()`, генерируются на стороне БД, а не в Python-коде, чтобы избежать рассинхронизации времени между инстансами приложения.
- Общие наборы полей у родственных сущностей (например, записи из разных внешних источников с одинаковой структурой) — выносятся в абстрактную базовую модель: `__abstract__ = True`, без `__tablename__`. Конкретные модели наследуются от неё и добавляют только то, что у них специфично (например, `updated_at` и `UniqueConstraint` только там, где это нужно). Это устраняет дублирование колонок без создания лишней таблицы.
- Таблицы-журналы событий (аналитика, аудит: "событие поиска", "сессия обработки", "событие изменения статуса") — отдельный, более простой тип модели: только `id`, `created_at` и набор полей-снимков на момент события. У них нет `updated_at` (запись неизменяема после создания), обычно не используется `doc=` (это не доменная сущность, которую разработчик долго изучает, а лог), а описание полей в `comment=` короче и фокусируется на "что значит это число/флаг" для аналитики. У репозитория такой модели нет методов `update_*`/`delete_*` — только `save_event`/`get_*` для агрегатов.
- Гибкие, не вполне табличные данные (произвольная структура ответа, список вопросов/ответов анкеты) — `JSONB`, а не отдельная связанная таблица, если данные всегда читаются целиком вместе с родительской записью и не требуют отдельных запросов/фильтрации по внутренним полям.

## 12. Pydantic-схемы

```python
from pydantic import BaseModel, ConfigDict, Field, field_validator


class ItemCreateRequest(BaseModel):
    """Тело запроса для создания новой записи Item."""

    model_config = ConfigDict(
        json_schema_extra={
            'example': {
                'external_id': 'ext-12345',
                'title': 'Пример названия',
            }
        }
    )

    external_id: str = Field(
        ...,
        description='Идентификатор сущности во внешней системе.',
        examples=['ext-12345'],
    )
    title: str = Field(
        ...,
        min_length=1,
        max_length=300,
        description='Название сущности.',
        examples=['Пример названия'],
    )


class ItemSchema(BaseModel):
    """Схема Item для ответов API — строится напрямую из ORM-объекта."""

    model_config = ConfigDict(from_attributes=True)

    id: int = Field(..., description='Уникальный идентификатор записи.', examples=[1])
    external_id: str = Field(..., description='Идентификатор сущности во внешней системе.', examples=['ext-12345'])
    title: str = Field(..., description='Название сущности.', examples=['Пример названия'])
    description: str = Field(
        '',
        description='Описание сущности.',
        examples=['Краткое описание записи.'],
    )

    @field_validator('description', mode='before')
    @classmethod
    def none_to_empty_string(cls, v: object) -> str:
        return '' if v is None else v


class ItemsListSchema(BaseModel):
    """Пагинированный список записей Item."""

    total: int = Field(..., description='Общее количество записей, удовлетворяющих фильтрам.', examples=[42])
    page: int = Field(..., description='Текущий номер страницы.', examples=[1])
    page_size: int = Field(..., description='Количество записей на странице.', examples=[10])
    items: list[ItemSchema] = Field(..., description='Список записей на текущей странице.')
```

Ключевые моменты этого примера:
- **Каждое поле — `Field(..., description=..., examples=[...])`**, даже там, где имя поля кажется самоочевидным — описание уходит прямо в OpenAPI/Swagger UI, а `examples` — в автогенерируемые примеры запросов/ответов.
- `model_config = ConfigDict(json_schema_extra={'example': {...}})` на схеме **запроса** — даёт цельный пример тела запроса в Swagger, а не только поэлементные примеры по каждому полю.
- `ConfigDict(from_attributes=True)` на схеме **ответа** — позволяет `ItemSchema.model_validate(orm_object)` напрямую, без промежуточного `.__dict__`/ручной сборки словаря.
- `@field_validator(..., mode='before')` — нормализация "грязных" значений из ORM/внешнего источника (например, `NULL` в колонке, которая по контракту API всегда строка) в момент валидации, а не точечными проверками в сервисе перед каждым возвратом схемы.
- Схема пагинированного списка (`ItemsListSchema`) всегда содержит `total`/`page`/`page_size`/`items` — единообразно для всех списочных эндпоинтов проекта.

Остальные правила:
- `@field_validator(..., mode='before')`, применённый сразу к нескольким полям одного типа одной декорацией (`@field_validator('field_a', 'field_b', mode='before')`), а не дублированием одинаковой проверки на каждое поле по отдельности.
- Одна и та же сущность в разных контекстах (краткий список / детальный просмотр / сохранённая выборка) — по возможности **одна общая схема**, а не три почти одинаковые. Разные представления получаются через опциональность полей с пустыми default'ами, а не через дублирование классов.
- Исключение из правила "одна схема" — когда публичное имя поля в API должно отличаться от внутреннего имени колонки в БД (например, БД хранит `legacy_code`, а контракт API отдаёт его как `category_code`, потому что для внешнего клиента это и есть осмысленное имя). В этом случае оправданно завести **две** схемы: публичную с `Field(..., validation_alias='legacy_code')` (переименование при чтении из ORM-объекта) для эндпоинтов, и внутреннюю с настоящими именами полей для сервисов/репозиториев, где переименование было бы только лишней путаницей. Критерий разделения — расхождение в именовании или составе полей между "что отдаём наружу" и "что используем внутри", а не "контекст использования" сам по себе.

## 13. Клиенты внешних API/LLM

- Один клиент = один внешний сервис, принимает готовый `httpx.AsyncClient` через конструктор, а не создаёт его сам — жизненный цикл самого `httpx.AsyncClient` (per-request или lifespan-managed) задаётся снаружи через DI (см. раздел 6, «Жизненный цикл HTTP-клиента»), клиент внешнего API от этого выбора не зависит.
- Низкоуровневый метод запроса (`_request_to_api`) централизует обработку HTTP/сетевых ошибок и возвращает унифицированную структуру `{"status": bool, "search_status": str, "response_data": dict}` — вызывающий код проверяет флаги, а не ловит исключения httpx напрямую.
- 404 от внешнего API — это не ошибка клиента, а доменный "не найдено" (`search_status="not_found"`), различается от реальных сбоев соединения/5xx.
- Повторные попытки (`_request_with_retry`) — отдельный метод-обёртка с константами `MAX_RETRIES`/`RETRY_DELAY` на уровне класса, с логированием каждой неуспешной попытки перед повтором.
- Ограничение конкурентности к внешнему API (когда у него есть rate limit по IP/ключу) — `asyncio.Semaphore` на уровне клиента, лимит — константа класса с комментарием, почему выбрано именно это значение.
- Параллельная загрузка многостраничных результатов — через `asyncio.gather` по страницам, с семафором на каждый запрос и явной агрегацией результатов после.
- По умолчанию старый код удаляется, а не оставляется закомментированным — для этого есть история git. Исключение: на проде стоит конкурентная реализация, она нестабильна именно из-за непредсказуемого поведения внешнего API (а не из-за бага в самом коде), и есть конкретный план вернуться к простой реализации в обозримый срок. Тогда временно допустим закомментированный блок с маркерами `--- SEQUENTIAL BACKUP ---` / `--- END ... ---` и явным объяснением, при каких условиях к нему возвращаться. Это исключение, а не норма — оставленный "на всякий случай" мёртвый код без срока жизни — это технический долг, а не страховка.
- Два уровня retry-стратегии в зависимости от характера внешнего сервиса:
  - **Фиксированная задержка** (`RETRY_DELAY` как константа, простой `await asyncio.sleep(RETRY_DELAY)` между попытками) — достаточно для внешних REST API, где сбои обычно кратковременны (сетевые моргания, временная перегрузка).
  - **Экспоненциальный backoff с джиттером** (`delay = min(max_delay, base_delay * 2**(attempt-1))`, плюс `± 10%` случайного отклонения) — для LLM/тяжёлых внешних сервисов, где параллельные клиенты, повторяющие запрос одновременно после фиксированной паузы, сами создают повторную перегрузку. Джиттер размывает повторные попытки во времени.
- Для LLM-клиентов retry-цикл различает два разных типа отказа и обрабатывает их одним циклом, но по сути разными причинами: **ошибка самого запроса** (сеть/HTTP/таймаут) и **ошибка контента ответа** (невалидный JSON, ответ не прошёл валидацию по Pydantic-схеме, пустой ответ). Оба типа уходят в retry, но логируются разными сообщениями, чтобы при чтении логов было понятно, рвётся транспорт или сама модель не может выдать нужный формат.
- Системные промпты для LLM (`prompt`, передаваемый как `system`-сообщение) — выносятся в отдельный модуль (`services/prompts/...py`) как именованные строковые константы, а не собираются инлайн в коде сервиса. Это отделяет "что мы просим у модели" (предмет частых правок, в том числе не-программистом) от "как мы это вызываем" (стабильный код).
- Если LLM поддерживает structured output по Pydantic-схеме — клиент валидирует ответ через `schema.model_validate_json(content)` сразу после извлечения текста, и ошибка валидации схемы — это повод для retry (модель могла просто плохо посчитать JSON), а не сразу финальный отказ.

## 14. Админка (sqladmin)

- `admin/__init__.py` — единая фабрика `create_admin(app, engine) -> Admin`, вызывается один раз из `main.py`. Здесь и только здесь собираются все `ModelView`/`BaseView`.
- `admin/auth.py` — `AuthenticationBackend` на логине и пароле из настроек (`admin_login`/`admin_password` как `SecretStr`), хранение факта входа в `request.session`. Никогда не использовать единый сервисный API-ключ как пароль в админку — это разные плоскости доступа с разным жизненным циклом.
- `admin/views.py` — каждая сущность представлена `ModelView`, со связкой:
  - `name`/`name_plural`/`icon` (FontAwesome) — обязательны, для визуальной узнаваемости разделов.
  - `column_list`/`column_searchable_list`/`column_sortable_list`/`column_default_sort` — явно перечислены, не оставлены на дефолт "все поля".
  - `column_formatters`/`column_formatters_detail` — кастомные рендеры (цветные бейджи статусов, pretty-print JSON, обрезка длинного текста) через небольшие функции `_fmt_*(model, attr) -> Markup`, переиспользуемые между списком и детальной страницей.
  - `can_create`/`can_edit`/`can_delete` — выставлены осознанно для каждой сущности по тому, что должно быть редактируемо человеком, а что — только просматриваемая система записей (логи событий, истории).
- Кастомные страницы, не привязанные к одной модели (агрегированная статистика, кастомные отчёты) — через `BaseView` + `@expose`, с собственным шаблоном Jinja, открывающие собственную короткую `AsyncSession` поверх переданного `engine` (а не через основной DI, потому что admin живёт вне обычного цикла запрос/Depends).
- Визуальный стиль админки соответствует общему дизайн-гайду проекта (тёмная тема, акцентный цвет для статусов/бейджей) — админка не выглядит как "техническая заглушка", а как часть продукта.

## 15. Миграции (Alembic)

- Имя файла миграции — `<дата>_<время>_<смысловое_описание>.py`, чтобы порядок и смысл были видны прямо в списке файлов без открытия содержимого.
- Одна миграция — одно концептуальное изменение (добавить таблицу / добавить поле / изменить constraint), не "накопить и сделать одним махом" — так проще откатывать точечно.
- `env.py` Alembic собирает `target_metadata` из той же `Base`, что и приложение — миграции и модели не могут разойтись по структуре без явного `autogenerate diff`.

## 16. Инфраструктура и запуск

- `Dockerfile` + `docker-compose.yml` + `entrypoint.sh` — entrypoint отвечает за ожидание готовности БД и прогон миграций перед стартом ASGI-сервера, а не приложение само пытается это сделать в коде.
- ASGI-сервер — Hypercorn/Uvicorn, конфигурация таймаутов (keep-alive, graceful timeout) синхронизирована по всей цепочке: nginx → ASGI-сервер → httpx-клиенты → таймауты LLM/внешних API. Рассинхронизация таймаутов на разных уровнях цепочки — частый источник трудноуловимых 504/502 под нагрузкой; при изменении таймаута на одном уровне нужно явно пересчитать соседние.
- `.env` / `.env.example` — `.env.example` всегда поддерживается актуальным (все переменные, которые есть в `Settings`, должны быть отражены), чтобы развернуть проект с нуля можно было без чтения кода настроек.

## 17. Стиль кода и документирования (сквозной)

- Аннотации типов — везде: параметры функций, возвращаемые значения, атрибуты класса. `str | None` вместо `Optional[str]` в новом коде (Python 3.10+), `Annotated[...]` для FastAPI-параметров.
- Докстринги — Google-стиль (`Args:`, `Returns:`, `Raises:`), по-русски, обязательны для всех публичных методов сервисов/репозиториев и для всех эндпоинтов. Для тривиальных приватных методов (одна строка смысла) — однострочный докстрин без секций.
- Комментарии в коде — только там, где не очевидно "почему" (выбор константы, временный compromise, нетривиальное ограничение внешнего API), никогда не пересказывают "что" делает следующая строка.
- Логические блоки внутри большого класса — разделяются комментарием-заголовком (`# Блок ...`), а не пустыми строками без подписи.
- Линтер — `ruff` (+ `.flake8` для совместимости), конфигурация лежит в корне проекта и применяется одинаково ко всем слоям без точечных исключений по папкам.

## 18. Тестирование

Тесты — обязательная часть архитектуры, не опциональная надстройка.

### Стратегия — пирамида тестов привязана к слоям

Слоистая архитектура (раздел 1) ценна тестами ровно настолько, насколько ею пользуются: каждый слой принимает зависимости через конструктор (раздел 6), значит каждый слой можно протестировать в изоляции, подсунув ему дублёр зависимости вместо реальной. Если этим не пользоваться, часть ценности архитектуры теряется — слои разделены, а проверяются всё равно только вручную через браузер/Postman.

| Слой | Тип теста | Что подменяется | Через что бежит |
|---|---|---|---|
| Сервис | unit | репозитории, клиенты, другие сервисы | `unittest.mock.AsyncMock(spec=...)` |
| Репозиторий | integration | ничего — реальная БД | `testcontainers` Postgres + реальная `AsyncSession` |
| Парсер/утилита | unit | ничего — чистая функция | прямой вызов с фикстурным словарём |
| Клиент внешнего API | unit | сетевой транспорт | `httpx.MockTransport` |
| Эндпоинт | integration (API-level) | сервис целиком (через `app.dependency_overrides`) | `httpx.AsyncClient` + `ASGITransport` |

Правило выбора: **репозиторий тестируется на реальной БД, а не на моках сессии.** Мокать `db_session.execute(...)` бессмысленно — такой тест проверяет, что мок вызван с нужными аргументами, а не что SQL-запрос на самом деле возвращает корректные данные (особенно с Postgres-специфичными вещами: `ilike`, `JSONB`, `ON DELETE CASCADE`, составные `UniqueConstraint` — на SQLite в памяти эти тесты будут лгать). `testcontainers` поднимает настоящий Postgres в Docker на время тестового прогона — это и есть единственный способ честно проверить репозиторий.

### Структура каталога

```
tests/
├── conftest.py                  # общие фикстуры: testcontainers, db_session, async_client
├── unit/
│   ├── services/                 # один файл = один сервис, зеркалит app/services/
│   └── utils/                     # парсеры, security.py и т.п.
├── integration/
│   └── repositories/              # один файл = один репозиторий, зеркалит app/repositories/
└── api/
    └── endpoints/                  # один файл = один роутер, зеркалит app/api/v1/endpoints/
```

Та же логика "один файл — одна сущность", что и в основном коде (раздел 2): тест на `ItemsService` лежит в `tests/unit/services/test_items_service.py`, тест на `ItemsRepository` — в `tests/integration/repositories/test_items_repository.py`.

### Фикстуры БД (`testcontainers` + реальный Postgres)

```python
# tests/conftest.py
import pytest_asyncio
from sqlalchemy.ext.asyncio import async_sessionmaker, create_async_engine
from testcontainers.postgres import PostgresContainer

from db.models.base import Base


@pytest_asyncio.fixture(scope="session")
def postgres_container():
    """Один контейнер Postgres на весь тестовый прогон — поднимать его на каждый тест слишком дорого."""
    with PostgresContainer("postgres:16-alpine") as container:
        yield container


@pytest_asyncio.fixture(scope="session")
async def engine(postgres_container):
    url = postgres_container.get_connection_url().replace("psycopg2", "asyncpg")
    engine = create_async_engine(url)
    async with engine.begin() as conn:
        await conn.run_sync(Base.metadata.create_all)
    yield engine
    await engine.dispose()


@pytest_asyncio.fixture
async def db_session(engine):
    """Сессия на отдельный тест с rollback после — тесты не видят данные друг друга."""
    session_factory = async_sessionmaker(engine, expire_on_commit=False)
    async with session_factory() as session:
        yield session
        await session.rollback()
```

- Контейнер — `scope="session"` (поднимается один раз на весь прогон, дорогая операция). Схема создаётся один раз через `Base.metadata.create_all` (для тестов alembic-миграции не прогоняются — это лишнее звено, схема и так строится из тех же моделей).
- Сессия — `scope="function"` (фикстура по умолчанию), с `rollback()` после каждого теста — изоляция между тестами без пересоздания контейнера.

### Unit-тест сервиса — зависимости подменены дублёрами

```python
# tests/unit/services/test_items_service.py
from unittest.mock import AsyncMock

import pytest

from exceptions.items import ItemNotFoundError
from repositories.items import ItemsRepository
from services.items import ItemsService


@pytest.mark.asyncio
async def test_get_item_raises_not_found_when_repository_returns_none():
    repository = AsyncMock(spec=ItemsRepository)
    repository.get_by_id.return_value = None
    service = ItemsService(items_repository=repository)

    with pytest.raises(ItemNotFoundError) as exc_info:
        await service.get_item(item_id="missing")

    assert exc_info.value.status_code == 404
```

- `AsyncMock(spec=ItemsRepository)` — `spec=` обязателен: без него мок тихо проглотит вызов несуществующего метода (например, после рефакторинга репозитория), и тест продолжит ложно зеленеть.
- Проверяется не просто факт исключения, а **конкретный тип и его `status_code`** — то, что реально долетит до клиента API через `raise HTTPException(status_code=error.status_code, ...)` в эндпоинте (раздел 7). Тест на "просто упало с ошибкой" не отличает 404 от 500.
- Сервис тестируется через его публичный интерфейс (`get_item`), не через приватные методы — приватные методы (`_normalize_*`, `_validate_*`) проверяются неявно, как часть сценария публичного метода, который их вызывает.

### Integration-тест репозитория — на реальной БД

```python
# tests/integration/repositories/test_items_repository.py
import pytest

from repositories.items import ItemsRepository


@pytest.mark.asyncio
async def test_save_and_get_by_external_id(db_session):
    repository = ItemsRepository(db_session)
    await repository.save_item({"external_id": "ext-1", "title": "Тестовая запись"})

    item = await repository.get_by_external_id(external_id="ext-1")

    assert item is not None
    assert item.title == "Тестовая запись"


@pytest.mark.asyncio
async def test_get_by_external_id_returns_none_when_not_found(db_session):
    repository = ItemsRepository(db_session)

    item = await repository.get_by_external_id(external_id="does-not-exist")

    assert item is None
```

Здесь не мокается ничего — проверяется реальный SQL-запрос на реальном движке Postgres. Это медленнее unit-тестов, но именно репозиторий — тот слой, где баг почти всегда в деталях конкретного диалекта БД (регистронезависимый поиск, типы колонок, поведение constraint'ов), а не в логике Python.

### Тест клиента внешнего API — подмена транспорта, не сети

```python
# tests/unit/clients/test_external_api_client.py
import httpx
import pytest

from clients.external_api_client import ExternalApiClient


@pytest.mark.asyncio
async def test_get_item_returns_not_found_on_404():
    def handler(request: httpx.Request) -> httpx.Response:
        return httpx.Response(404)

    httpx_client = httpx.AsyncClient(transport=httpx.MockTransport(handler))
    client = ExternalApiClient(httpx_client=httpx_client)

    result = await client.get_item(item_id="missing")

    assert result["search_status"] == "not_found"
```

`httpx.MockTransport` подменяет именно транспортный слой httpx, а не сам клиент — код клиента (`_request_to_api`, обработка `httpx.HTTPStatusError`) реально выполняется и проверяется, без обращения к сети.

### API-тест эндпоинта — через `dependency_overrides`

```python
# tests/api/endpoints/test_items.py
from unittest.mock import AsyncMock

import pytest
from httpx import ASGITransport, AsyncClient

from dependencies.services import get_items_service
from exceptions.items import ItemNotFoundError
from main import app
from services.items import ItemsService


@pytest.fixture(autouse=True)
def clear_overrides():
    yield
    app.dependency_overrides.clear()


@pytest.mark.asyncio
async def test_get_item_endpoint_returns_404_with_detail():
    fake_service = AsyncMock(spec=ItemsService)
    fake_service.get_item.side_effect = ItemNotFoundError(item_id="missing")
    app.dependency_overrides[get_items_service] = lambda: fake_service

    async with AsyncClient(transport=ASGITransport(app=app), base_url="http://test") as client:
        response = await client.get("/api/v1/items/missing")

    assert response.status_code == 404
    assert "missing" in response.json()["detail"]
```

- `app.dependency_overrides[get_items_service] = ...` подменяет ровно ту фабрику, что зарегистрирована в `dependencies/services.py` (раздел 6) — благодаря тому, что в эндпоинте используется именно type alias `ItemsServiceDep`, а не ручной вызов сервиса, подмена работает без единой правки в коде эндпоинта.
- Этот уровень теста не проверяет бизнес-логику (она уже проверена unit-тестами сервиса) — он проверяет **контракт HTTP**: правильный код ответа, что `detail` долетел до тела ответа, что роутинг и сериализация `response_model` работают. Дублировать здесь все ветки бизнес-логики сервиса — не нужно.
- `clear_overrides` автоиспользуемая фикстура — без неё override, выставленный в одном тесте, протечёт в следующий и даст ложный результат.

### Соглашения

- Имя теста — `test_<метод_или_сценарий>_<условие>_<ожидаемый_результат>` (например, `test_get_item_raises_not_found_when_repository_returns_none`) — по названию понятно, что именно проверяется, без чтения тела теста.
- Структура теста — Arrange / Act / Assert, разделённые пустой строкой, без явных комментариев-разделителей (структура должна быть видна по форме, не по подписям).
- Не тестируется то, что гарантирует сама библиотека (Pydantic корректно валидирует типы, SQLAlchemy корректно строит SQL для `select(...)`) — тестируется код проекта: ветвление, обработка ошибок, маппинг исключений, граничные случаи бизнес-правил (TTL истёк/не истёк, все источники недоступны одновременно, дубликат vs новая запись).
- Каждое новое доменное исключение (раздел 8) — минимум один тест, что сервис/репозиторий действительно его поднимает в нужном сценарии, и (на уровне API-теста) что оно мапится в правильный `status_code`.
- CI обязан гонять тесты **до** шага деплоя, а не параллельно или после — деплой не должен запускаться на сломанных тестах. Тестовый шаг в CI закладывается с первого коммита, а не добавляется постфактум, когда проект уже вырос.
